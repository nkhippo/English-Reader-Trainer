## 1. アプリの思想（何のために作るか）

### 1.1 解こうとしている課題

英語の語彙・コロケーション・チャンクは、文法規則だけでは身につかない **個別頻度由来の知識** である。フラッシュカードでも放任読書でもなく、**「異なる文脈で、間隔をあけて、多数回出会わせる」** ことで自動化を目指す。

### 1.2 6つの設計柱

| # | 柱 | 意味 |
|---|-----|------|
| 1 | **単位＝3〜6文の易しいパッセージ** | 目標チャンクを1つ（パッセージあたり2〜4個）自然に織り込む。周囲は既知語で固める（i+1） |
| 2 | **文脈内・間隔再会エンジン** | 各チャンクが 1日後・3日後・7日後… と **新しい文脈で** 再登場する |
| 3 | **翻訳を挟ませない** | 意味は文脈から取る。日本語訳はタップで一時表示する **補助** のみ |
| 4 | **低摩擦・高流量 UX** | スワイプで読み流す。採点や診断の摩擦を作らない |
| 5 | **軽い検索だけ差す** | たまに Cloze 的な想起を起こす（将来） |
| 6 | **卒業＝遭遇回数 × 文脈数** | 正答率より **異なる文脈での遭遇数** を価値指標とする |

### 1.3 Listening Trainer との関係

| | Listening Trainer | English Reader Trainer |
|---|-------------------|------------------------|
| 鍛える回路 | 音→意味 | 形→意味（語彙・チャンクの自動化） |
| 入力 | 音声中心 | 視覚中心（音声は補助） |
| 難度設計 | 難・少・反復・診断 | 易・多・没入・露出 |
| 共通資産 | CEFR データ、Drive 音声キャッシュ、Claude 生成のノウハウ | 同左 |

設計原則が **正反対** なので別アプリ。CEFR 語彙・チャンク JSON は Drive `shared/` で共有する。

### 1.4 UX のメタファ

> **書籍の余白に書き込まれた marginalia（注釈）**

学習アプリというより **読書体験**。チャンクの説明はポップアップではなく、右余白（モバイルでは下から）に滑り込む marginalia として表示する。

## 2. 現状の仕様（思想をどう実装しているか）

Phase 1〜3 まで実装済み。以下は **2026年6月時点の実装** と、仕様書との **差分** を含む。

### 2.1 アーキテクチャ

```
ブラウザ (React / GitHub Pages)
    ↕ GET/POST（GAS Web App）
Google Apps Script
    ↕
Google Sheets（構造化データ） + Google Drive（JSON / 音声）
    ↕
Claude API（翻訳バッチ・将来はパッセージ生成）
```

**設計上の核心（仕様書 §2.2）**

1. Sheets と Drive を役割分離 — Sheets はクエリ可能な行データ、Drive はファイル
2. **SRS エンジンは GAS 側** — クライアントは「次に何を浴びるべきか」を決めない
3. 1パッセージに複数チャンク（2〜4個）— 新規露出と既習想起の混在
4. 正答率ではなく **異なる文脈での遭遇数** を重視

### 2.2 データモデル（4シート + Drive）

| リソース | 役割 | 現状 |
|----------|------|------|
| `chunks_master` | CEFR 語彙・チャンク（約 7,125 件） | ✅ 投入済み。`ja_translation` はバッチ生成中または未完了の行あり |
| `user_progress` | チャンクごとの SRS 状態 | ✅ Got it / Still hard / passive で更新 |
| `encounter_log` | 遭遇イベント（追記専用） | ✅ 記録中 |
| `passages_meta` | 生成パッセージのメタデータ | ⬜ Phase 4 で本格利用（現状ほぼ未使用） |
| Drive `shared/` | `cefr_words.json`, `cefr_chunks.json` | ✅ |
| Drive `passages/` | パッセージ本文 JSON | ⬜ Phase 4 |
| Drive `audio/` | TTS キャッシュ | ⬜ Phase 5 |

### 2.3 SRS エンジン（Phase 3 — 実装済み）

**ステージと間隔**

| stage | 次回まで | 状態イメージ |
|-------|----------|--------------|
| 0 | 即座 | 未遭遇（new） |
| 1 | +1日 | 1度遭遇 |
| 2 | +3日 | 2度遭遇 |
| 3 | +7日 | 3度遭遇 |
| 4 | +14日 | 4度遭遇 |
| 5 | +30日 | graduated 相当 |

**評価シグナル**

| 操作 | signal | 効果 |
|------|--------|------|
| 「理解した →」 | `got_it` | stage +1 |
| 「まだ難しい」 | `still_hard` | stage −1（最低 0） |
| 30秒無操作 | `passive` | stage 維持、+1日 |
| （将来）スキップ | `skipped` | 維持 |

**卒業条件**

```
encounter_count >= 5
AND distinct_passages_count >= 3
AND still_hard_count / encounter_count < 0.3
→ status = graduated
```

**GAS エンドポイント（実装済み）**

| action | 用途 |
|--------|------|
| `session` | 起動時：パッセージ + ヘッダー統計を1回で取得 |
| `due_chunks` | 期限到来・新規チャンクの取得（**バックエンドのみ。フロント未接続**） |
| `generate_passage` | パッセージ取得（**現状は固定テンプレート**） |
| `log_encounter` | 遭遇ログ + `user_progress` 更新 |
| `update_progress` | SRS 更新のみ |
| `stats` | CEFR 帯別の reviewing / graduated 集計 |

### 2.4 フロントエンド（Phase 1〜3 — 実装済み）

| 機能 | 仕様書の意図 | 現状 |
|------|--------------|------|
| パッセージ表示 | 動的生成された読み物 | **CEFR 帯ごと固定テンプレ 3 本**（計9パターン） |
| チャンクハイライト | 本文中のターゲットを強調 | ✅ `{{chunk}}` マークアップから表示 |
| Marginalia | 右余白に chunk 情報 | ✅ 日本語訳・例文・遭遇回数・stage |
| 翻訳オーバーレイ | 本文タップで一時表示 | ✅ 約3.5秒 |
| Got it / Still hard | 低摩擦フィードバック | ✅ |
| CEFR 切替 | A1+A2 / B1 / B2 | ✅ |
| ヘッダー統計 | reviewing / graduated | ✅ `stats` から取得 |
| スワイプ・キーボード | 読み流し | ✅ |

**現状の限界（Phase 4 前）**

- パッセージ内容が **固定** のため、同じ文脈の繰り返ししかできない
- `distinct_passages_count` がテンプレ 3 本の範囲で頭打ちになり、**卒業判定の「3つの異なる文脈」が実質満たせない**
- `due_chunks` は実装されているが、**パッセージ選定に使われていない**
- ユーザの SRS 状態が読み物の内容に **反映されない**

### 2.5 手動運用タスク（開発外）

| タスク | 状態 |
|--------|------|
| `importChunksFromCefr()` | ✅ 完了（7,125 件） |
| `enrichAllTranslations(10)` を `remaining: 0` まで | 🔄 要確認・継続 |
| GAS 再デプロイ（コード変更のたび） | 運用中 |
| 実際に読んで SRS データを蓄積 | 🔄 推奨（後述） |

## 3. Phase 4 で対応しようとしていること

### 3.1 一言で言うと

> **「SRS が選んだチャンクを、毎回ちがう新しい文脈のパッセージに載せて再会させる」**

現状は **エンジン（SRS）と読み物（固定テンプレ）が切り離されている**。Phase 4 はこの2つを接続するフェーズである。

### 3.2 なぜ Phase 4 が必要か（思想とのギャップ）

| 設計柱 | Phase 3 まで | Phase 4 以降 |
|--------|--------------|--------------|
| 文脈内・間隔再会 | 間隔は記録するが **文脈は同じ3本** | **毎回新しいパッセージ** で再会 |
| 異なる文脈での遭遇数 | `distinct_passages_count` が増えない | 生成パッセージごとに増える |
| 卒業判定 | 条件を満たしにくい | 意味のある卒業が可能 |
| サーバが次を決める | due は計算するが **使っていない** | due から選んで Claude が文を組む |

### 3.3 Phase 4 の具体タスク（仕様書 §6）

1. **`/due_chunks` 連携**
   - 期限到来チャンク + 新規チャンクから 2〜4 個を選ぶ
   - 混在ルール例：new 1 + learning 1〜2 + reviewing 1

2. **Claude パッセージ生成**
   - CEFR 語彙制約（i+1）
   - 指定チャンクを **自然に** 埋め込む（詰め込み禁止）
   - 英文 + 日本語訳 + チャンク位置（`char_start` / `char_end`）を JSON で出力

3. **バリデーション & 永続化**
   - 指定チャンクが本文に含まれるか検証、失敗時は再生成
   - Drive `passages/` に JSON 保存
   - `passages_meta` に登録

4. **フロント接続**
   - 固定テンプレではなく API 生成パッセージを表示
   - 同一ユーザ・同一日内の重複出題を避ける

### 3.4 Acceptance（完了条件）

- [ ] 起動時に **動的生成** されたパッセージが表示される
- [ ] 各パッセージに 2〜4 ターゲットチャンク
- [ ] ハイライト位置がズレない
- [ ] 同じ日に同じパッセージが繰り返し出ない

## 4. 「1〜2週間、実際に読む」と Phase 4 の関係

Phase 4 の **実装前提** ではなく、**運用上の推奨** である。

| 目的 | 説明 |
|------|------|
| SRS の動作確認 | Got it / Still hard / passive が意図通り `user_progress` に反映されるか |
| データの蓄積 | `encounter_log` に実データが溜まり、Phase 4 実装後の動作検証に使える |
| 利用感の把握 | 固定テンプレでも読み流し UX・Marginalia の使い勝手を確認できる |

Phase 4 は固定テンプレがなくても **開発自体は開始できる**。ただし、due チャンクの選定ロジックを検証するには、`user_progress` にある程度の行があるとよい。

## 5. 今後の構想（Phase 5〜6 およびスコープ外）

### Phase 5: TTS 音声統合（補助）

**目的**: 読みながら聴ける選択肢（dual coding）。必須機能ではない。

- `/tts` — Listening Trainer と **Drive キャッシュ共有**
- パッセージ音声 → `passages_meta.audio_drive_url`
- フロントに再生 UI

**前提**: Phase 4 でパッセージが Drive に保存されていること。

### Phase 6: 進捗ダッシュボード

**目的**: 接触量・卒業数の可視化（AI 推薦はしない）。

- CEFR × ステータス（new / learning / reviewing / graduated）の内訳
- 日次接触カレンダー
- 「卒業近い」「停滞中」チャンクリスト
- 累積読書時間

**現状**: ヘッダーに reviewing / graduated の **件数のみ** 表示（Phase 6 の縮小版）。

### スコープ外（仕様書 §8 より）

- フラッシュカード型 UI
- 文法解説・総合テスト
- 多ユーザ認証（現状 `user_id = naoya` 固定）
- AI による学習計画の自動推薦

## 7. 次に取るべきアクション（推奨順）

### 開発前（任意・短期）

1. `enrichAllTranslations(10)` を `remaining: 0` まで実行
2. アプリで数日〜1〜2週間読み、SRS データを蓄積
3. `user_progress` / `encounter_log` が期待通り増えることを確認

### 開発（Phase 4）

1. GAS: `due_chunks` → チャンク選定 → Claude 生成 → バリデーション → Drive/Sheets 保存
2. GAS: `generate_passage` をテンプレから動的生成に差し替え
3. フロント: 生成パッセージのハイライト位置対応
4. 受け入れ条件（§3.4）の確認

### その後

- Phase 5（音声）→ Phase 6（ダッシュボード）の順が自然
- Listening Trainer の Drive キャッシュ完成度に応じて Phase 5 のタイミングを調整

## 8. 用語集

| 用語 | 意味 |
|------|------|
| チャンク | 語彙・コロケーション・句動詞などの学習単位（例: `managed to`） |
| パッセージ | 3〜6文の短い読み物。2〜4 チャンクを含む |
| 遭遇（encounter） | パッセージを読み、チャンクに触れたイベント |
| SRS | Spaced Re-encounter — 間隔を空けた **文脈内** 再会 |
| due | `next_due_at` を過ぎ、再投入すべきチャンク |
| graduated | 卒業 — 十分な遭遇と文脈多様性を満たした状態 |
| marginalia | 右余白のチャンク注釈パネル |
